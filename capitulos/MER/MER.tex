\chapter{Modelo Entidade-Relacionamento (MER)}

Antes de iniciar a criação de um banco de dados, é necessário compreender como as informações estão organizadas na vida real. Para isso, utiliza-se o \textbf{Modelo Entidade-Relacionamento (MER)}, que é uma abordagem conceitual e lógica para abstrair os dados de um sistema e suas regras de negócio. A representação visual e gráfica desse modelo é feita através do \textbf{Diagrama Entidade-Relacionamento (DER)}. Neste capítulo, será adotada especificamente a Notação de Peter P. Chen.

O MER ajuda a equipe de desenvolvimento a identificar pilares fundamentais do sistema:
\begin{itemize}
    \item Quais informações o sistema precisa guardar;
    \item Quais são os principais objetos (entidades) do sistema;
    \item As características (atributos) de cada objeto;
    \item Como esses objetos se conectam entre si (relacionamentos).
\end{itemize}

Importa destacar que a negligência em relação a uma modelagem adequada frequentemente resulta em sistemas rígidos, com redundância de dados e inconsistências que prejudicam a tomada de decisão organizacional. Para evitar tais problemas, o desenvolvimento de um banco de dados deve ser compreendido como um processo cíclico e dividido em etapas bem definidas, garantindo que a tecnologia atenda às reais necessidades do usuário \cite{angelotti2010banco}.

A etapa inicial, foco deste capítulo, é a \textbf{Modelagem Conceitual}. Nela, o objetivo primordial é identificar quais dados devem ser persistidos e como eles interagem entre si, estabelecendo uma visão abstrata da base de dados. Para que este modelo seja fidedigno, o projetista deve realizar um levantamento de requisitos — frequentemente via entrevistas — para mapear as expectativas e os objetivos do usuário final \cite{angelotti2010banco}.

Sequencialmente, avança-se para a \textbf{Modelagem Lógica}, detalhada no Capítulo \ref{cap:modelo_logico}. Neste estágio, o foco desloca-se para a estruturação sistêmica das informações: o modelo conceitual é traduzido para um formato compatível com a tecnologia de implementação. No contexto de bancos de dados relacionais, adota-se o modelo relacional e aplicam-se técnicas de normalização para mitigar redundâncias, conforme será discutido no Capítulo \ref{cap:normalizacao} \cite{angelotti2010banco}.

Por fim, ocorre a \textbf{Modelagem Física}, tratada no Capítulo \ref{cap:modelo_fisico}. É neste momento que o projeto se materializa no ambiente computacional, definindo-se tipos de dados, alocação de espaço em disco, criação de índices e restrições de armazenamento. Com a estrutura validada, a base de dados é efetivamente instanciada em um Sistema Gerenciador de Banco de Dados (SGBD), tarefa que exige domínio da linguagem SQL e das especificidades técnicas do software escolhido \cite{angelotti2010banco}.

\section{Aspectos Históricos}

Historicamente, o Modelo Entidade-Relacionamento (MER) foi formalizado em 1976 por Peter P. Chen em seu célebre artigo \textit{"The Entity-Relationship Model - Toward a Unified View of Data"} \cite{peterchen}. 

Antes da consolidação desta abordagem, o projeto de bancos de dados baseava-se predominantemente em modelos hierárquicos e de rede, cujas estruturas eram excessivamente rígidas, complexas e de difícil compreensão para profissionais que não detivessem profundo conhecimento técnico.

É importante notar que, à época da formalização do MER, o cenário tecnológico já havia sido revolucionado pela proposta do Modelo Relacional, apresentado por Edgar F. Codd em 1970 \cite{codd1970relational}. No entanto, o modelo de Codd, embora matematicamente rigoroso e superior aos antigos modelos hierárquicos e de rede, carecia de uma forma semântica e visual para representar as regras de negócio antes da implementação em tabelas.

A grande inovação de Chen residiu na percepção de que a modelagem de dados seria mais eficiente se espelhasse a percepção humana do mundo real \cite{peterchen}. Ele propôs que a realidade fosse mapeada como um conjunto de \textbf{entidades} (coisas ou conceitos abstratos), que possuem \textbf{atributos} (características) e que estabelecem \textbf{relacionamentos} entre si de maneira lógica e natural \cite{peterchen}.

Atualmente, este modelo permanece como o padrão global para a fase inicial de qualquer projeto de banco de dados. Sua representação gráfica funciona como uma linguagem universal, eliminando barreiras de comunicação entre desenvolvedores e \textit{stakeholders}.

\subsection{Entidades}

No MER, uma \textbf{Entidade} representa qualquer coisa do mundo real (seja ela física ou abstrata) sobre a qual o sistema precisa armazenar informações.

As entidades costumam ser divididas em dois tipos:
\begin{itemize}
    \item \textbf{Físicas (Concretas):} São coisas tangíveis, que existem no mundo material, como clientes, produtos, fornecedores, transportadoras ou armazéns.
    \item \textbf{Lógicas (Abstratas):} São conceitos, eventos ou transações que existem de forma lógica, a exemplo de pedidos, pagamentos, entregas, categorias de produtos e carrinhos de compras.
\end{itemize}

A técnica mais simples para identificar entidades durante a leitura das regras de negócio (ou entrevistas com o cliente) é a análise gramatical, na qual foca-se nos substantivos. Se a palavra nomeia algo e precisa-se guardar várias informações sobre ela para que o sistema funcione, então, encontrou-se uma entidade. Por exemplo, considere a seguinte regra de negócio: 

\begin{quote}
\textit{"O \texttt{cliente} realiza um \texttt{pedido} contendo vários \texttt{produtos}."}
\end{quote}

Os termos destacados (cliente, pedido e produto) são substantivos sobre os quais faz-se necessário guardar detalhes (como o nome do cliente, a data do pedido e o preço do produto). Logo, são excelentes candidatos a entidades. No diagrama, essas entidades serão desenhadas utilizando o símbolo do retângulo.

Por fim, é de suma importância estabelecer a distinção conceitual entre uma entidade e a sua instância. A entidade representa a classe, o "molde" ou a estrutura geral (por exemplo, a entidade Produto). Por outro lado, a \textbf{instância} é a ocorrência real, preenchida e específica daquele molde (por exemplo, o produto "Notebook Dell, ID 5678"). Durante a fase de modelagem de dados e na elaboração do DER, modelam-se exclusivamente as entidades, e não as suas instâncias individuais.

\subsection{Atributos}
Uma vez definidas as entidades do sistema, o próximo passo é detalhar as informações que as compõem. Os atributos são, portanto, os dados específicos a serem armazenados sobre uma entidade. Na entidade cliente, por exemplo, os atributos podem ser o seu nome, endereço de e-mail e CPF. 

Um conceito fundamental associado aos atributos é o seu \textbf{domínio}. O domínio representa o conjunto de todos os valores válidos e possíveis que aquele campo pode assumir. Por exemplo, o domínio do atributo "Preço" em um sistema de vendas não aceitará valores negativos.

Para refinar a modelagem, os atributos são classificados em categorias de acordo com o seu comportamento, conforme listado a seguir:

\subsubsection{Atributo Simples e Monovalorado}
A grande maioria dos atributos do dia a dia se encaixa simultaneamente nestas duas definições. Eles representam o padrão básico de armazenamento:
\begin{itemize}
    \item \textbf{Simples (ou Atômico):} É aquele que não pode ser subdividido, representando uma informação básica e direta.
    \item \textbf{Monovalorado:} É aquele concebido para guardar apenas um único valor por vez em cada registro.
\end{itemize}
Como ilustração, pode-se citar o CPF, a data de nascimento e o nome completo de um cliente, visto que são dados que não se dividem em partes menores e cada instância possui apenas um registro válido de cada.

A idade de um cliente ilustra bem esse cenário, pois pode ser calculada dinamicamente pelo sistema com base na data de nascimento vinculada àquele registro. \subsection{Chave Primária (Primary Key - PK)}

Dentre todos os atributos de uma entidade, um recebe um papel de extrema relevância: a \textbf{Chave Primária}. Sua função exclusiva é identificar cada registro de forma única e inequívoca dentro do banco de dados, garantindo que não existam duas entradas idênticas.

Para ser considerada uma chave primária, o atributo deve obedecer a duas regras fundamentais:
\begin{itemize}
    \item \textbf{Unicidade}: O valor não pode se repetir em hipótese alguma (como a impossibilidade de registrar dois produtos diferentes com o mesmo código de barras no sistema).
    \item \textbf{Obrigatoriedade}: O campo não pode ser nulo, pois todo registro precisa ter sua identificação estabelecida.
\end{itemize}

A escolha da chave primária deve ser feita com cautela, priorizando atributos estáveis, que não sofram alterações ao longo do tempo, evitando impactos na integridade referencial do sistema. Além disso, em cenários mais complexos, pode-se optar pela utilização de identificadores artificiais (como IDs numéricos gerados automaticamente), especialmente quando não há um atributo natural suficientemente confiável para exercer tal função.

\subsubsection{Tipos de Chave Primária}
A chave primária pode ser classificada conforme a quantidade de atributos necessários para identificar unicamente cada ocorrência de uma entidade. Em termos práticos, ela pode ser formada por apenas um atributo ou pela combinação de dois ou mais atributos, a depender da estrutura dos dados e das regras de negócio do sistema modelado. Assim, quanto à sua composição, a chave primária divide-se em dois tipos principais: 

\begin{enumerate}
    \item \textbf{Simples}: Formada por apenas um único atributo, como o CPF de um cliente, o ID de um pedido ou o código SKU de um produto.
    \item \textbf{Composta}: Formada pela união de dois ou mais atributos. É utilizada em situações específicas onde um único campo isolado não é capaz de garantir a unicidade do registro.
\end{enumerate}

É crucial destacar um erro conceitual muito comum durante o aprendizado da modelagem de dados: uma entidade jamais possuirá "duas chaves primárias". O conceito de chave composta significa que existe uma única chave primária, mas ela necessita da combinação de dois ou mais atributos trabalhando simultaneamente em bloco para garantir a exclusividade daquele registro. No exemplo ilustrado a seguir, a entidade não possui duas chaves soltas; ela possui apenas uma chave primária, que é formada pelo par contendo o ID do Produto em conjunto com o ID do Armazém.

No Diagrama Entidade-Relacionamento, o atributo escolhido como chave primária é tradicionalmente destacado com o seu nome \underline{sublinhado}.

\subsection{Relacionamentos}

As entidades em um banco de dados não existem isoladas, elas interagem constantemente para refletir as regras de negócio do mundo real. Essa interação estrutural entre duas ou mais entidades é o que se chama de \textbf{Relacionamento}.

De forma análoga à técnica utilizada para as entidades, a maneira mais simples de identificar relacionamentos durante a análise gramatical dos requisitos é focar nos verbos. Se a palavra indica uma ação, um evento ou uma associação entre as entidades já identificadas, então encontrou-se um relacionamento. Por exemplo, considere a seguinte regra de negócio:

\begin{quote}
\textit{"Uma transportadora \texttt{entrega} vários pedidos."}
\end{quote}

O termo destacado (entrega) é um verbo que demonstra como a entidade Transportadora interage com a entidade Pedido dentro do sistema. Logo, é o candidato exato para ser o relacionamento. No Diagrama Entidade-Relacionamento, essas conexões são desenhadas visualmente utilizando o símbolo do losango, sendo unidas às entidades por meio de linhas retas.

Por sua vez, para modelar essas conexões corretamente e evitar falhas estruturais na base de dados, analisa-se a dinâmica da interação por meio de duas propriedades essenciais: o grau e a cardinalidade.

\subsubsection{Grau do Relacionamento}

O grau de um relacionamento indica quantas entidades participam de uma determinada associação. Os relacionamentos binários conectam duas entidades distintas e são os mais comuns nos projetos de modelagem de dados.

Entretanto, há também os relacionamentos unários, também chamados de autorrelacionamentos, nos quais uma entidade se relaciona com ela própria. Um exemplo típico ocorre em estruturas hierárquicas, como categorias de produtos, em que uma categoria pode estar vinculada a outra como categoria superior.

Além disso, existem os relacionamentos ternários ou superiores, que representam associações mais complexas, nas quais três ou mais entidades participam simultaneamente de uma mesma interação ou transação.

Apesar de a notação prever relacionamentos ternários ou de graus ainda maiores (genericamente chamados de relacionamentos n-ários), a sua utilização direta na modelagem não é recomendada. Relacionamentos n-ários aumentam excessivamente a complexidade visual do modelo, dificultam a correta definição das cardinalidades e não possuem uma conversão direta e intuitiva para tabelas na fase de modelagem lógica, que será debatida futuramente neste livro.

Para contornar essa complexidade, a boa prática exige a decomposição do relacionamento n-ário. O procedimento consiste em transformar o próprio relacionamento em uma nova entidade (frequentemente denominada entidade associativa) e conectá-la às entidades originais exclusivamente por meio de relacionamentos binários. 

Essa abordagem oferece duas vantagens principais:

\begin{itemize}
    \item \textbf{Atributos Próprios:} Permite registrar dados específicos da transação, como a data da entrega e o custo do frete, que não pertenciam isoladamente a nenhuma das outras três entidades.
    \item \textbf{Simplificação Lógica:} Facilita a futura conversão para o modelo relacional (tabelas), pois o banco de dados lidará apenas com associações binárias simples.
\end{itemize}

No exemplo ilustrado anteriormente, em vez de manter um único losango conectando as três frentes, a modelagem ideal consistiria em criar uma entidade chamada \texttt{Fornecimento}. Essa nova entidade assumiria relacionamentos binários independentes com o Fornecedor, o Produto e o Armazém, simplificando a estrutura e permitindo o armazenamento de atributos próprios do fornecimento (como data de entrega e custo de frete).

\subsubsection{Cardinalidade}

Independentemente do grau, os relacionamentos são restritos e quantificados de acordo com sua \textbf{cardinalidade}. Na notação de Peter Chen, essas regras de proporção são frequentemente detalhadas expressando-se o par \textbf{(mínimo, máximo)} ao lado das linhas de conexão.

\begin{itemize}
    \item \textbf{Cardinalidade Mínima (Opcionalidade):} Indica o limite inferior de participações da entidade no relacionamento. Geralmente, assume os valores 0 (participação opcional) ou 1 (participação obrigatória). Contudo, pode ser qualquer número inteiro exigido pela regra de negócio (por exemplo, um produto deve ter no mínimo 2 fotos cadastradas).
    \item \textbf{Cardinalidade Máxima:} Indica o limite superior de participações da entidade. Costuma assumir o valor 1 (no máximo uma) ou "n" (várias participações sem limite predefinido). Assim como a mínima, também pode ser um valor numérico exato fixado pela regra (por exemplo, um cliente pode salvar no máximo 3 cartões de crédito em sua conta).
\end{itemize}

Embora as combinações mais frequentes na literatura sejam \textbf{(0,1)}, \textbf{(1,1)}, \textbf{(0,n)} ou \textbf{(1,n)}, as regras de uma organização podem estabelecer limites perfeitamente específicos, resultando em cardinalidades como \textbf{(2,5)} (no mínimo dois, no máximo cinco) ou \textbf{(4,4)} (exatamente quatro itens obrigatórios).

Ao classificar o relacionamento de forma macro, observando apenas as cardinalidades máximas envolvidas na associação, identificam-se os seguintes tipos clássicos:

\begin{itemize}
    \item \textbf{1:1 (Um para um)}: Ocorre quando um registro de uma entidade se relaciona a apenas um registro de outra. É o caso de um pedido e a sua respectiva Nota Fiscal (considerando que cada pedido gera apenas uma nota e cada nota refere-se a apenas um pedido).
    \item \textbf{1:N (Um para muitos)}: Ocorre quando um registro de uma entidade pode se relacionar a vários registros de outra. É o exemplo de um cliente que pode realizar vários pedidos ao longo do tempo, mas cada pedido pertence a apenas um cliente.
    \item \textbf{N:N (Muitos para muitos)}: Ocorre quando múltiplos registros de uma entidade se relacionam com múltiplos registros de outra. No e-commerce, um pedido contém vários produtos e, simultaneamente, o mesmo produto pode estar presente em centenas de pedidos diferentes.
\end{itemize}

\subsection{Entidade Fraca}

Para finalizar os conceitos fundamentais do diagrama, destaca-se a \textbf{Entidade Fraca}. Como o nome sugere, é uma entidade que não possui força para existir sozinha no banco de dados, sendo totalmente dependente de outra entidade (chamada de entidade forte).

\textbf{Características Principais:}
\begin{itemize}
    \item Não possui uma chave primária própria capaz de identificá-la globalmente.
    \item Sua existência está condicionada: se a entidade forte vinculada a ela for apagada do sistema, a entidade fraca perde o sentido e deve ser apagada junto (exclusão em cascata).
\end{itemize}

Em um e-commerce, um exemplo clássico é o \textbf{Item de Pedido}. Um registro que diz "3 unidades do produto X" não faz sentido sozinho no banco de dados; ele precisa estar obrigatoriamente vinculado a um \textbf{Pedido} específico. Se o pedido for cancelado e excluído do sistema, todos os itens que dependiam dele também sairão do sistema automaticamente.

No diagrama de Chen, a entidade fraca é representada de forma destacada por um \textit{retângulo com borda dupla}, e seu relacionamento associado também recebe a borda dupla.
