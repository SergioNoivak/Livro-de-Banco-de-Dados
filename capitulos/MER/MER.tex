\chapter{Modelo Entidade-Relacionamento (MER)}

Antes de começar a criar as tabelas de um banco de dados, precisamos entender como as informações estão organizadas na vida real. Para isso, utilizamos o \textbf{Modelo Entidade-Relacionamento (MER)}, que é uma forma visual de representar os dados de um sistema através do \textbf{Diagrama Entidade-Relacionamento (DER)}.

O DER ajuda a equipe a identificar:
\begin{itemize}
    \item Quais informações o sistema precisa guardar;
    \item Quais são os principais objetos (entidades) do sistema;
    \item As características (atributos) de cada objeto;
    \item Como esses objetos se conectam entre si (relacionamentos).
\end{itemize}

\section{Aspectos Históricos}

O MER foi criado em 1976 por \textbf{Peter P. Chen}. Ele percebeu que a melhor forma de modelar um banco de dados era olhar para o mundo real como um conjunto de \textbf{objetos} (entidades) que possuem \textbf{características} (atributos) e que se \textbf{relacionam} entre si.

Hoje, esse modelo é o padrão mundial para o início de qualquer projeto de banco de dados, pois facilita a conversa entre os desenvolvedores e os clientes.

\section{Tipos de Atributos}

Os atributos são as informações que queremos guardar sobre uma entidade. Por exemplo, na entidade \textbf{Aluno}, os atributos podem ser: Nome, E-mail e Matrícula. Existem 6 categorias principais de atributos:

\subsection{1. Atributo Simples}
É aquele que não pode ser dividido. Ele representa uma informação básica e direta.
\textbf{Exemplos:} CPF, RG, Sexo, Peso.

\subsection{2. Atributo Composto}
É um atributo que pode ser dividido em partes menores para facilitar o uso. O exemplo clássico é o endereço.
\textbf{Exemplo:} O \textbf{Endereço} pode ser dividido em: Rua, Número, Bairro e Cidade.

\subsection{3. Atributo Monovalorado}
É o atributo que guarda apenas \textbf{um valor} por vez. 
\textbf{Exemplo:} Uma pessoa tem apenas um \textbf{CPF} e apenas uma \textbf{Data de Nascimento}.

\subsection{4. Atributo Multivalorado}
É quando uma entidade pode ter \textbf{vários valores} para o mesmo atributo. No diagrama, ele é representado por uma \textbf{elipse dupla}.
\textbf{Exemplo:} Uma pessoa pode ter vários \textbf{Telefones} ou vários \textbf{E-mails}.

\subsection{5. Atributo Nulo (NULL)}
Ocorre quando um campo fica vazio. Isso acontece se a informação for desconhecida ou se não se aplicar àquela pessoa específica.
\textbf{Exemplo:} O campo "Complemento" de um endereço pode ser nulo se a pessoa morar em uma casa sem complemento.

\subsection{6. Atributo Derivado (Calculado)}
É um atributo que não precisa ser digitado, pois pode ser calculado a partir de outro. No diagrama, ele é representado por uma \textbf{elipse tracejada}.
\textbf{Exemplo:} A \textbf{Idade} pode ser calculada a partir da Data de Nascimento.

\section{Chave Primária (Primary Key - PK)}

A \textbf{Chave Primária} é o atributo mais importante de uma entidade. Ela serve para identificar cada registro de forma única, garantindo que não existam dois registros iguais.

\textbf{Regras da Chave Primária:}
\begin{itemize}
    \item \textbf{Unicidade}: O valor não pode se repetir (ex: não podem existir dois alunos com o mesmo CPF).
    \item \textbf{Obrigatoriedade}: O campo não pode ser nulo (\texttt{NOT NULL}).
\end{itemize}

No diagrama (DER), a Chave Primária é identificada com o nome \textbf{\underline{sublinhado}}.

\subsection{Tipos de Chave Primária}

\begin{enumerate}
    \item \textbf{Simples}: Formada por apenas um atributo. Ex: CPF, Matrícula.
    \item \textbf{Composta}: Formada por dois ou mais atributos juntos. Isso acontece quando um único campo não é suficiente para identificar o registro.
\end{enumerate}

\section{Relacionamentos}

Os \textbf{relacionamentos} mostram como as entidades interagem entre si. Eles são classificados pela sua \textbf{cardinalidade} (quantidade de ligações):

\begin{itemize}
    \item \textbf{1:1 (Um para um)}: Uma pessoa tem um único CPF.
    \item \textbf{1:N (Um para muitos)}: Uma mãe pode ter vários filhos, mas cada filho tem apenas uma mãe biológica.
    \item \textbf{N:N (Muitos para muitos)}: Um aluno pode estudar várias disciplinas, e uma disciplina pode ter vários alunos.
\end{itemize}

\section{Entidade Fraca}

Uma \textbf{entidade fraca} é aquela que não consegue ser identificada sozinha. Ela depende de outra entidade (entidade forte) para existir.

\textbf{Características:}
\begin{itemize}
    \item Não possui chave primária própria.
    \item Se a entidade forte for apagada, ela também deixa de fazer sentido.
\end{itemize}

\textbf{Exemplo:} Um \textbf{Dependente} só existe no sistema se estiver ligado a um \textbf{Funcionário}. Se o funcionário for demitido, os seus dependentes saem do sistema com ele.

No diagrama, a entidade fraca é representada por um \textbf{retângulo com borda dupla}.

\section{Atividade Prática: Sistema de Fazenda}

\subsection{Contexto}

Uma fazenda leiteira em Rio Verde deseja modernizar o controle de sua produção. Atualmente, os registros são manuais, o que dificulta a gestão.

\begin{center}
    \includegraphics[width=9cm]{fazenda.png}
    \captionof{figure}{Fazenda em Rio Verde (GO).}
\end{center}

\subsection{Requisitos}

\begin{itemize}
    \item \textbf{Vacas}: Devem ter um código único, nome, data de nascimento, raça e peso.
    \item \textbf{Funcionários}: Identificados por código e nome. Eles realizam as ordenhas.
    \item \textbf{Ordenha}: Registro de cada vez que uma vaca é ordenhada. Deve guardar a Data, Hora e Quantidade de leite (litros).
    \item \textbf{Veterinário}: Atende as vacas quando necessário. Deve guardar o código, nome e CRMV.
\end{itemize}

\subsection{Regras de Negócio}

\begin{itemize}
    \item Uma vaca pode ser ordenhada várias vezes ao longo do tempo.
    \item Cada ordenha é feita por um funcionário.
    \item Um veterinário pode atender várias vacas, e uma vaca pode receber vários atendimentos.
\end{itemize}

\subsection{O que fazer?}

Com base nas informações acima, desenhe o \textbf{Diagrama Entidade-Relacionamento (DER)}:
\begin{enumerate}
    \item Identifique as \textbf{Entidades} (Vacas, Funcionários, etc.).
    \item Defina os \textbf{Atributos} e a \textbf{Chave Primária} de cada uma.
    \item Desenhe os \textbf{Relacionamentos} e defina as \textbf{Cardinalidades} (1:1, 1:N ou N:N).
\end{enumerate}

\begin{observacao}
\textbf{Instruções de Entrega:}
O Diagrama Entidade-Relacionamento deve ser feito de forma legível (à mão ou ferramenta digital como brModelo/Lucidchart). O arquivo final deve ser convertido para \textbf{PDF} e conter apenas o diagrama finalizado.
\end{observacao}

\subsection{Critérios de Avaliação}
A avaliação considerará a correta identificação das entidades, adequação das chaves primárias, precisão das cardinalidades frente às regras de negócio e a organização visual do diagrama.
