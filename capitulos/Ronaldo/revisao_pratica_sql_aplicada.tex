\chapter{Revisão Prática da Parte III: SQL Aplicada}
\label{cap:revisao_pratica_sql_aplicada}

A Parte III conduziu o leitor do aprofundamento em SQL até temas de administração inicial de banco de dados. Ao longo das Aulas 10 a 17, foram estudados os principais grupos da linguagem SQL, a definição e modificação de estruturas, a manipulação de dados, a construção de consultas analíticas, o uso de views, o controle de acesso, os índices, as transações e as funções.

Este capítulo final tem como objetivo consolidar esse percurso. Em vez de introduzir um novo bloco teórico isolado, a proposta é revisar, integrar e aplicar os conteúdos da Parte III em um encadeamento prático. Assim, o leitor pode perceber como os diversos recursos estudados se articulam em um cenário mais próximo do uso real de um banco de dados relacional.

\section{Panorama de fechamento da Parte III}

Ao final desta parte, o estudante atingiu a maturidade técnica para compreender a SQL como linguagem definitiva de implementação e uso de bancos relacionais. O percurso prático permitiu dominar a criação de estruturas consolidadas com DDL e a manipulação fluida de dados com DML. Avançando na complexidade, tornou-se possível elaborar consultas analíticas sofisticadas utilizando agregações, junções e subconsultas, bem como refinar esses retornos com paginação, remoção de duplicidades e funções de janela. Além do puro manuseio de dados, o estudo englobou abstrações arquiteturais, como o uso de \texttt{VIEW}s para simplificar relatórios, e políticas fundamentais de segurança com a gestão de roles e privilégios. Por fim, o ciclo se completou com noções de desempenho através de índices, proteção de operações vitais usando transações e o encapsulamento de lógicas internas por meio de funções estruturadas. Essa bagagem mostra que a Parte III não se limita a “comandos de SQL”, mas constitui um bloco real de transição entre a teoria de banco de dados e a engenharia de software aplicada.

\section{Mapa conceitual resumido}

Uma forma útil de visualizar a Parte III é organizar seus tópicos em quatro grandes eixos de responsabilidade. O primeiro é o \textbf{Eixo estrutural}, que abarca a modelagem física com comandos de \texttt{CREATE}, \texttt{ALTER} e \texttt{DROP}, gerenciando diretamente a tipagem de dados, as restrições de integridade e a chave estrangeira. O segundo é o \textbf{Eixo operacional}, voltado à manutenção do dia a dia do banco de dados, utilizando intensivamente os comandos básicos de \texttt{INSERT}, \texttt{SELECT}, \texttt{UPDATE} e \texttt{DELETE}. O terceiro bloco é o \textbf{Eixo analítico}, que eleva a consulta simples para o nível gerencial; aqui o profissional aplica junções (\texttt{JOIN}), agregações com \texttt{GROUP BY} e \texttt{HAVING}, subconsultas, operadores de ordenação, unicidade, união e funções de janela, além de criar \texttt{VIEW}s. O quarto e último é o \textbf{Eixo administrativo}, que lida com o comportamento sistêmico do servidor: controle de acesso via \texttt{GRANT} e \texttt{REVOKE}, otimização com índices e a garantia atômica das transações.

\section{Estudo de caso integrador}

Para consolidar a Parte III, será utilizado um pequeno sistema acadêmico contemplando as tabelas de cursos, alunos, disciplinas, notas, turma e matrícula. A ideia não é esgotar todas as possibilidades do sistema, mas mostrar como os diversos recursos se combinam em uma sequência coerente.

\section{Passo 1 -- Estrutura inicial}

\textbf{Criação de cursos:}
\begin{sqlcode}
CREATE TABLE cursos (
    curso_id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    coordenador VARCHAR(100)
);
\end{sqlcode}

\textbf{Criação de alunos:}
\begin{sqlcode}
CREATE TABLE alunos (
    aluno_id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    idade INTEGER,
    curso_id INTEGER,
    status VARCHAR(20) DEFAULT 'ATIVO',
    FOREIGN KEY (curso_id) REFERENCES cursos(curso_id)
);
\end{sqlcode}

\textbf{Criação de disciplinas:}
\begin{sqlcode}
CREATE TABLE disciplinas (
    disciplina_id SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL
);
\end{sqlcode}

\textbf{Criação de notas:}
\begin{sqlcode}
CREATE TABLE notas (
    nota_id SERIAL PRIMARY KEY,
    aluno_id INTEGER NOT NULL,
    disciplina_id INTEGER NOT NULL,
    nota NUMERIC(5,2) NOT NULL,
    FOREIGN KEY (aluno_id) REFERENCES alunos(aluno_id),
    FOREIGN KEY (disciplina_id) REFERENCES disciplinas(disciplina_id)
);
\end{sqlcode}

\textbf{Criação de turma:}
\begin{sqlcode}
CREATE TABLE turma (
    turma_id SERIAL PRIMARY KEY,
    nome VARCHAR(50) NOT NULL,
    vagas_disponiveis INTEGER NOT NULL DEFAULT 30
);
\end{sqlcode}

\textbf{Criação de matrícula:}
\begin{sqlcode}
CREATE TABLE matricula (
    matricula_id SERIAL PRIMARY KEY,
    aluno_id INTEGER NOT NULL,
    turma_id INTEGER NOT NULL,
    FOREIGN KEY (aluno_id) REFERENCES alunos(aluno_id),
    FOREIGN KEY (turma_id) REFERENCES turma(turma_id)
);
\end{sqlcode}

\section{Passo 3 -- Consultas operacionais}

\textbf{Listando todos os alunos cadastrados:}
\begin{sqlcode}
SELECT *
FROM alunos;
\end{sqlcode}

\textbf{Filtrando apenas os alunos com status ativo:}
\begin{sqlcode}
SELECT nome, idade
FROM alunos
WHERE status = 'ATIVO';
\end{sqlcode}

\textbf{Atualizando o status de um registro específico:}
\begin{sqlcode}
UPDATE alunos
SET status = 'INATIVO'
WHERE aluno_id = 5;
\end{sqlcode}

\textbf{Removendo uma nota lançada indevidamente:}
\begin{sqlcode}
DELETE FROM notas
WHERE nota_id = 3;
\end{sqlcode}

Essas operações retomam diretamente a manipulação diária do sistema (DML), cobrindo a entrada, alteração e deleção de registros fundamentais, conforme estudado nas aulas anteriores.

\section{Passo 4 -- Consultas analíticas}

\textbf{Quantidade de alunos por curso:}
\begin{sqlcode}
SELECT
    curso_id,
    COUNT(*) AS total_alunos
FROM alunos
GROUP BY curso_id;
\end{sqlcode}

\textbf{Cursos com pelo menos dois alunos matriculados:}
\begin{sqlcode}
SELECT
    curso_id,
    COUNT(*) AS total_alunos
FROM alunos
GROUP BY curso_id
HAVING COUNT(*) >= 2;
\end{sqlcode}

\textbf{Buscando alunos cuja idade está acima da média geral:}
\begin{sqlcode}
SELECT nome, idade
FROM alunos
WHERE idade > (
    SELECT AVG(idade)
    FROM alunos
);
\end{sqlcode}

\textbf{Gerando um relatório integrado de aluno, disciplina e nota:}
\begin{sqlcode}
SELECT a.nome AS aluno,
       d.nome AS disciplina,
       n.nota
FROM notas n
INNER JOIN alunos a
    ON n.aluno_id = a.aluno_id
INNER JOIN disciplinas d
    ON n.disciplina_id = d.disciplina_id;
\end{sqlcode}

\textbf{Identificando cursos que não possuem nenhum aluno:}
\begin{sqlcode}
SELECT c.nome
FROM cursos c
LEFT JOIN alunos a
    ON c.curso_id = a.curso_id
WHERE a.aluno_id IS NULL;
\end{sqlcode}

Aqui aparecem, integradamente, as agregações, subconsultas e os diferentes tipos de junções, retomando o núcleo analítico de SQL.

\section{Passo 5 -- Refinamento das consultas}

\textbf{Ordenação primária e secundária por curso e nome:}
\begin{sqlcode}
SELECT nome, curso_id
FROM alunos
ORDER BY curso_id, nome;
\end{sqlcode}

\textbf{Limitando o retorno para a lógica de paginação na aplicação:}
\begin{sqlcode}
SELECT nome, curso_id
FROM alunos
ORDER BY nome
LIMIT 3 OFFSET 0;
\end{sqlcode}

\textbf{Recuperando apenas cursos distintos entre os alunos:}
\begin{sqlcode}
SELECT DISTINCT curso_id
FROM alunos
ORDER BY curso_id;
\end{sqlcode}

\textbf{União de dados entre alunos e coordenadores:}
\begin{sqlcode}
SELECT nome, 'ALUNO' AS origem
FROM alunos
UNION ALL
SELECT coordenador, 'COORDENADOR' AS origem
FROM cursos;
\end{sqlcode}

\textbf{Cálculo da média de idade do curso mantendo o detalhamento por aluno:}
\begin{sqlcode}
SELECT
    nome,
    curso_id,
    idade,
    AVG(idade) OVER (PARTITION BY curso_id) AS media_curso
FROM alunos;
\end{sqlcode}

\section{Passo 6 -- Criação de abstrações com Views}

\textbf{View para simplificar o relatório acadêmico de notas:}
\begin{sqlcode}
CREATE VIEW vw_notas_alunos AS
SELECT
    a.nome AS aluno,
    d.nome AS disciplina,
    n.nota
FROM notas n
INNER JOIN alunos a
    ON n.aluno_id = a.aluno_id
INNER JOIN disciplinas d
    ON n.disciplina_id = d.disciplina_id;
\end{sqlcode}

Uma vez criada, a consulta complexa pode ser executada rapidamente com um \texttt{SELECT * FROM vw\_notas\_alunos ORDER BY aluno, disciplina;}, demonstrando como o banco centraliza lógicas que seriam exaustivas para a aplicação cliente.

\section{Passo 7 -- Controle de acesso básico}

\textbf{Criando um perfil genérico de leitura para relatórios:}
\begin{sqlcode}
CREATE ROLE leitura_academica;
GRANT SELECT ON alunos TO leitura_academica;
GRANT SELECT ON cursos TO leitura_academica;
GRANT SELECT ON vw_notas_alunos TO leitura_academica;
\end{sqlcode}

\textbf{Criando um usuário real e vinculando-o ao perfil criado:}
\begin{sqlcode}
CREATE ROLE usuario_consulta LOGIN PASSWORD 'consulta123';
GRANT leitura_academica TO usuario_consulta;
\end{sqlcode}

Esse trecho recupera o conceito de segurança, evidenciando que a disponibilização de informações confidenciais deve ser sempre mediada por políticas estritas de acesso.

\section{Passo 8 -- Otimização com Índices}

\textbf{Acelerando as buscas frequentes pelo nome do aluno:}
\begin{sqlcode}
CREATE INDEX idx_alunos_nome
ON alunos(nome);
\end{sqlcode}

\textbf{Acelerando os filtros combinados nas verificações de matrícula:}
\begin{sqlcode}
CREATE INDEX idx_matricula_aluno_turma
ON matricula(aluno_id, turma_id);
\end{sqlcode}

Esses índices são justificados diretamente pelos padrões de consulta plausíveis no sistema do dia a dia, melhorando drasticamente o desempenho de sistemas em produção.

\section{Passo 9 -- Integridade com Transações}

\textbf{Bloco transacional para matricular aluno e subtrair vaga atomicamente:}
\begin{sqlcode}
BEGIN;

INSERT INTO matricula (aluno_id, turma_id)
VALUES (1, 1);

UPDATE turma
SET vagas_disponiveis = vagas_disponiveis - 1
WHERE turma_id = 1;

COMMIT;
\end{sqlcode}

A transação protege uma operação logicamente composta. Sem ela, um problema de rede ou desligamento do servidor poderia registrar a matrícula do aluno sem atualizar corretamente a vaga disponível, corrompendo a consistência do negócio.

\section{Passo 10 -- Funções de apoio no banco}

\textbf{Criando uma rotina reutilizável para contagem de matrículas:}
\begin{sqlcode}
CREATE OR REPLACE FUNCTION total_matriculas_aluno(p_aluno_id INTEGER)
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_total INTEGER;
BEGIN
    SELECT COUNT(*)
    INTO v_total
    FROM matricula
    WHERE aluno_id = p_aluno_id;

    RETURN v_total;
END;
$$;
\end{sqlcode}

Essa função, que pode ser invocada via \texttt{SELECT total\_matriculas\_aluno(1);}, ilustra perfeitamente como o banco de dados pode oferecer cálculos automatizados e verificações padronizadas para as mais variadas plataformas externas.

\section{Erros clássicos de integração e Boas Práticas}

Ao revisar toda a Parte III, alguns erros aparecem como especialmente recorrentes na curva de aprendizagem e devem ser ativamente evitados:

\textbf{Esquecer a cláusula WHERE em UPDATE ou DELETE:} Isso fatalmente alterará ou removerá todas as linhas da tabela, causando perdas incalculáveis em produção. 

\textbf{Esquecer a condição de junção em um JOIN:} A falta de um \texttt{ON} ou de uma chave estrangeira válida faz com que o banco gere um produto cartesiano gigantesco, multiplicando dados indevidamente.

\textbf{Confundir WHERE com HAVING:} É vital lembrar que o \texttt{WHERE} atua como um filtro individual de linhas antes da agregação, enquanto o \texttt{HAVING} filtra unicamente os blocos que já foram agrupados pelo \texttt{GROUP BY}.

\textbf{Não usar transação em operação composta:} Modificar o saldo financeiro de duas contas (transferência) sem encapsular a operação em um \texttt{BEGIN} e \texttt{COMMIT} abre margem para dinheiro "desaparecer" se houver uma interrupção súbita no meio do processo.

Para que seu projeto relacional alcance maturidade, cultive o hábito de validar operações destrutivas escrevendo primeiramente uma consulta simples (\texttt{SELECT}). Trate o banco de dados como um ambiente ativo de regras de negócio, não o limitando a um mero depósito mudo de arquivos. 

\begin{quadro_dica}
Uma maneira madura de pensar SQL é enxergar cada comando em três planos ao mesmo tempo: o que ele faz, que risco ele envolve e que papel ele desempenha no ecossistema do seu sistema corporativo.
\end{quadro_dica}
