\chapter{Introdução à SQL}
\label{cap:introducao_sql}

A linguagem SQL (\textit{Structured Query Language}) ocupa posição central no desenvolvimento de bancos de dados relacionais. Depois de compreender conceitos introdutórios, modelagem conceitual, modelo lógico e normalização, o estudante precisa transformar estruturas abstratas em comandos concretos executáveis por um sistema gerenciador de banco de dados. É exatamente nessa transição que a SQL se torna indispensável.

Esta aula tem como objetivo introduzir a linguagem SQL como instrumento de definição, manipulação, consulta, controle e administração inicial de dados. Mais do que apresentar comandos isolados, a proposta é mostrar a SQL como elo entre o projeto do banco e a sua implementação prática.

\section{A passagem do modelo para a implementação}

Nas etapas anteriores do projeto de banco de dados, o foco esteve voltado para perguntas fundamentais sobre a modelagem da informação. A primeira grande questão foi determinar quais dados devem ser armazenados. Essa análise exigiu um esforço de abstração para identificar as entidades relevantes e seus respectivos atributos, separando informações essenciais para o negócio de dados desnecessários ou redundantes. O objetivo foi assegurar que a base teórica contemplasse tudo o que o sistema precisa para operar.

A segunda pergunta essencial investigou como esses dados se relacionam. Por meio de modelos conceituais, buscou-se compreender as associações, dependências e cardinalidades que unem entidades distintas. Esse mapeamento estrutural foi indispensável para transformar dados isolados em uma rede coesa de conhecimento, prevendo como as informações interagiriam e seriam cruzadas no futuro.

A terceira pergunta voltou-se para as regras de integridade, questionando quais restrições precisam ser respeitadas. A modelagem não se preocupou apenas com o formato dos dados, mas também com a definição de limites de valores, chaves identificadoras, regras de unicidade e obrigatoriedades. Identificar essas restrições ainda no plano conceitual é uma medida preventiva crucial para evitar a entrada de dados inválidos ou inconsistentes.

Por fim, a última pergunta estruturante buscou resolver como representar corretamente a realidade do domínio. O grande desafio dessa fase foi garantir que o modelo abstrato fosse um reflexo fiel, lógico e semântico do mundo real, traduzindo adequadamente os processos, o vocabulário corporativo e as dinâmicas operacionais do negócio em estruturas consistentes.

Essas perguntas são fundamentais, mas ainda pertencem ao plano de modelagem. Quando o projeto avança para a fase prática de implementação, surgem novos desafios de ordem estritamente técnica. O desenvolvedor passa a ter que lidar com a infraestrutura de dados reais, precisando descobrir como criar o banco de dados propriamente dito no SGBD escolhido, como definir estruturalmente cada tabela e suas respectivas colunas de acordo com as restrições físicas. Além disso, torna-se necessário estabelecer os métodos para a inserção, atualização e remoção diária da informação, criar rotinas eficientes para consultar cruzamentos de dados complexos e, sobretudo, configurar as permissões para preservar o acesso e a consistência. É para orquestrar todas essas respostas em um ambiente computacional que a SQL foi concebida.

\section{O que é SQL e seu caráter declarativo}

A SQL é a linguagem universalmente adotada para a interação com bancos de dados relacionais. Embora seu nome — \textit{Structured Query Language} — destaque a realização de consultas como sua função principal, a SQL evoluiu para cobrir todo o ciclo de vida do dado. De forma prática, a linguagem é o instrumento que não só cria e altera o layout das tabelas no disco, mas também permite popular essas tabelas com inserções e atualizações, filtrar registros de forma dinâmica, configurar políticas rigorosas de acesso e administrar transações atômicas de negócio.

Uma das características mais marcantes e vantajosas da SQL é o seu paradigma declarativo. Em linguagens de programação imperativas, como Python ou C, o desenvolvedor é forçado a escrever o passo a passo lógico detalhando "como" o computador deve chegar a um resultado. Na SQL, o desenvolvedor se preocupa exclusivamente em expressar a intenção final, ou seja, "o que" ele deseja obter. Ao estruturar uma consulta simples solicitando os nomes de todos os alunos pertencentes a um determinado curso, o usuário não se preocupa em programar laços de repetição ou navegar pelos endereços de memória. O SGBD assume toda essa complexidade nos bastidores, tomando decisões otimizadas sobre qual o melhor índice a consultar ou a melhor estratégia para percorrer as tabelas. Essa abstração foi e continua sendo um dos pilares do sucesso massivo dos bancos relacionais.

\section{SQL padrão versus dialetos}

Existe um esforço contínuo do ANSI e da ISO para manter a SQL sob uma padronização universal. No entanto, o mercado de software e a concorrência por melhor desempenho fizeram com que cada grande fabricante desenvolvesse o próprio SGBD com características singulares. Surgiram assim os dialetos da SQL. Sistemas amplamente conhecidos, como PostgreSQL, MySQL, SQL Server e Oracle Database, compartilham o mesmo núcleo sintático básico — todos compreendem comandos como SELECT, UPDATE ou DELETE de forma quase idêntica. Contudo, assim que a exigência técnica se aprofunda, as diferenças emergem claramente no comportamento das transações, no gerenciamento de concorrência, nas funções auxiliares e até mesmo na nomenclatura oficial dos tipos de dados suportados. Essa diversidade reforça a ideia de que o aprendizado do banco de dados relacional deve focar nos conceitos e arquiteturas essenciais, permitindo que o aluno se adapte facilmente ao dialeto exigido pelo cenário profissional em que estiver inserido.

\section{As sublinguagens da SQL e a complementaridade dos capítulos}

No decorrer deste e dos próximos capítulos, perceberemos que a SQL não é um amontoado caótico de instruções. Para fins didáticos e organizacionais, a linguagem é dividida em grupos, ou sublinguagens, focadas em responsabilidades estritas no ciclo de vida do banco.

O primeiro bloco lógico que constrói as fundações do sistema é conhecido como \textbf{DDL} (\textit{Data Definition Language}). A DDL atua quase como um "arquiteto", fornecendo comandos como \texttt{CREATE}, \texttt{ALTER} e \texttt{DROP} para levantar paredes e divisórias lógicas, moldando os bancos, schemas, tabelas e restrições.

Uma vez que as fundações estão prontas, entram em cena os dois núcleos operacionais vitais onde desenvolvedores gastam a maior parte de seu tempo. O primeiro deles é a \textbf{DML} (\textit{Data Manipulation Language}), responsável por nutrir as tabelas. É por meio dos comandos de manipulação como \texttt{INSERT}, \texttt{UPDATE} e \texttt{DELETE} que registros são cadastrados, os preços são corrigidos e o cotidiano da empresa efetivamente ocorre no banco. O segundo núcleo, inseparável e muitas vezes o mais complexo intelectualmente, é a \textbf{DQL} (\textit{Data Query Language}), dominada primariamente pela cláusula \texttt{SELECT}. A DQL é a porta de saída da informação; é ela que possibilita cruzar milhares de linhas em milissegundos, gerar relatórios agregados com agrupamentos e extrair o conhecimento estratégico de volta para a aplicação. Nos capítulos que se seguem — especialmente sob a condução das aulas dedicadas pela autoria do Professor Leandro —, você fará um extenso mergulho prático e dirigido sobre como dominar a inserção e extração sofisticada dessas informações (DML e DQL).

Por fim, os bancos corporativos contam com a \textbf{DCL} (\textit{Data Control Language}), focada em segurança e permissões através do uso de \texttt{GRANT} e \texttt{REVOKE}, bem como a \textbf{TCL} (\textit{Transaction Control Language}), responsável pela integridade durante cenários complexos de concorrência com as instruções de salvamento e reversão de tarefas (\texttt{COMMIT} e \texttt{ROLLBACK}).

\subsection{A armadilha clássica: DDL versus DML em provas e concursos}

Um dos erros mais comuns cometidos por estudantes --- e frequentemente explorado em provas do ENADE e exames de concursos públicos --- é a confusão entre a estruturação do banco de dados e a manipulação de seu conteúdo. É fundamental ter clareza de que não é possível inserir um dado em uma estrutura que não existe. A Linguagem de Definição de Dados (DDL) atua como o arquiteto e o pedreiro da obra, erguendo as paredes e definindo os espaços (as tabelas e colunas). Sem essa etapa prévia, qualquer tentativa de usar a Linguagem de Manipulação de Dados (DML), como uma instrução de inserção ou atualização, resultará em um erro imediato do sistema gerenciador, pois o dado não encontrará seu respectivo contêiner físico.

\section{SGBD, banco, schema e objetos}

Em um ambiente relacional de grande porte, a organização vai além de tabelas isoladas soltas no vácuo. No ponto mais alto da hierarquia, o SGBD representa o motor principal que coordena tudo — seja um servidor PostgreSQL na nuvem ou instalado localmente. Dentro desse motor, podem existir vários bancos de dados completamente independentes, sendo um para cada software da empresa. Mergulhando mais fundo, cada banco de dados pode ser dividido em compartimentos lógicos chamados schemas, que agrupam sob o mesmo teto áreas do negócio (por exemplo, um schema para recursos humanos e outro para o setor financeiro). Finalmente, é dentro dos schemas que residem os objetos finais da SQL: as tabelas, as visões, as funções geradoras de sequências e os índices. Compreender essa árvore arquitetural ajuda a evitar problemas sérios de direcionamento e perda de dados durante a manutenção rotineira de um servidor empresarial.

\section{Comparação entre SQL e linguagem de programação}

É extremamente comum que alunos busquem comparar o estudo da SQL com o aprendizado de linguagens puras de propósito geral como Python, Java ou C++. Embora partilhem a lógica computacional, possuem finalidades diametralmente distintas. Uma linguagem de programação clássica coordena o fluxo da interface, as decisões baseadas em variáveis globais, os laços de repetição em memória e as requisições de rede. A SQL, em contraste, não foi criada para desenhar janelas de aplicativo ou programar robôs, mas sim para ser a forma mais rápida, segura e escalável de gerenciar conjuntos de informações persistentes. Em quase todo projeto comercial moderno, ambas caminham juntas: enquanto o Python interage com o usuário e constrói a regra de negócio da plataforma, ele simultaneamente envia solicitações em SQL para delegar a custódia das informações ao banco de dados.

\section{Boas práticas iniciais de escrita SQL}

Mesmo nestes passos iniciais, estabelecer hábitos de escrita coesa evita o surgimento de códigos legados problemáticos. Um bom script em SQL deve ser, antes de tudo, legível e organizado. O uso de nomes significativos para as tabelas e colunas, evitando abreviações obscuras que dificultam a leitura no futuro, é essencial. Existe ainda uma convenção muito difundida no meio acadêmico e em fóruns profissionais: embora o interpretador do banco de dados geralmente ignore diferenças entre letras maiúsculas e minúsculas nas instruções da linguagem, é extremamente recomendável escrever as palavras reservadas do sistema (como \texttt{SELECT}, \texttt{FROM}, \texttt{WHERE}) sempre em caixa alta, mantendo nomes de campos e tabelas em caixa baixa. Isso oferece um contraste visual automático, agilizando enormemente o processo de leitura e depuração do código por outros membros da equipe.
