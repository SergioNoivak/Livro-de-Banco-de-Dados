\chapter{Refinamento de Consultas e Views}
\label{cap:refinamento_consultas_views}

À medida que o domínio da SQL avança, o estudante percebe que não basta apenas selecionar colunas, aplicar filtros e juntar tabelas. Em aplicações reais, torna-se necessário refinar resultados, controlar ordenação, limitar a quantidade de linhas devolvidas, eliminar duplicidades, combinar consultas distintas, produzir análises por grupo sem perder a granularidade das linhas e encapsular consultas complexas para reutilização.

Esta aula aprofunda exatamente esse nível de sofisticação. O foco recai sobre cláusulas e recursos avançados do \texttt{SELECT}, bem como sobre o conceito de \texttt{VIEW}, que permite transformar consultas recorrentes em objetos lógicos reutilizáveis dentro do banco de dados.

\section{Por que refinar consultas}

No uso cotidiano de bancos de dados corporativos, demandas analíticas complexas surgem a todo momento. O cientista de dados ou engenheiro de software frequentemente precisa mostrar registros em uma ordem muito específica, paginar listas extensas para consumo em APIs, extrair apenas valores distintos para popular menus (\textit{dropdowns}), ou ainda combinar resultados oriundos de fontes completamente diferentes. Além disso, análises de mercado exigem comparar cada linha com o desempenho médio de seu próprio grupo sem perder a granularidade dos dados, exigindo também a criação de objetos virtuais para simplificar a manutenção de relatórios repetitivos.

Tais necessidades exigem que o comando \texttt{SELECT} deixe de ser apenas uma ferramenta de extração simples para atuar como um sofisticado mecanismo de organização, síntese analítica e formatação direta de apresentação dos dados.

\section{ORDER BY}

A cláusula \texttt{ORDER BY} define a ordem em que as linhas serão apresentadas. Sem ela, não se deve assumir que os resultados surgirão sempre na mesma sequência.

\subsection{Ordenação simples}

\begin{sqlcode}
SELECT nome, idade
FROM alunos
ORDER BY nome;
\end{sqlcode}

Nesse caso, os alunos serão exibidos em ordem alfabética pelo nome.

\subsection{Ordenação decrescente}

\begin{sqlcode}
SELECT nome, idade
FROM alunos
ORDER BY idade DESC;
\end{sqlcode}

Agora, os alunos são listados da maior para a menor idade.

\subsection{Ordenação por múltiplos critérios}

\begin{sqlcode}
SELECT nome, curso, idade
FROM alunos
ORDER BY curso ASC, idade DESC;
\end{sqlcode}

Essa consulta ordena primeiro pelo curso e, dentro de cada curso, coloca os alunos mais velhos antes dos mais jovens.

\section{LIMIT}

A cláusula \texttt{LIMIT} restringe a quantidade de linhas retornadas pela consulta.

\begin{sqlcode}
SELECT *
FROM alunos
ORDER BY aluno_id
LIMIT 5;
\end{sqlcode}

A imposição de limites é uma tática indispensável para a preservação de recursos do servidor de banco de dados e da rede. Ao limitar a saída, o administrador consegue exibir rapidamente uma pequena amostra exploratória, impedindo o travamento de ferramentas de terminal ao inspecionar tabelas com bilhões de registros. Além de economizar processamento durante fases de testes empíricos, essa técnica é o núcleo fundamental dos algoritmos de paginação em aplicações web, onde os registros são fatiados e entregues ao cliente sob demanda, viabilizando relatórios executivos rápidos e interfaces fluidas.

\section{OFFSET}

A cláusula \texttt{OFFSET} determina quantas linhas iniciais serão ignoradas antes de retornar o conjunto desejado.

\begin{sqlcode}
SELECT *
FROM alunos
ORDER BY aluno_id
LIMIT 5 OFFSET 5;
\end{sqlcode}

Nesse caso, o banco ignora as primeiras cinco linhas e devolve as cinco seguintes.

\subsection{Paginação}

\texttt{LIMIT} e \texttt{OFFSET} são frequentemente usados juntos para paginação.

Por exemplo, para processar um relatório dividido em lotes de dez registros, a primeira página executará \texttt{LIMIT 10 OFFSET 0}. Ao avançar na leitura, a interface requisitará a página seguinte informando \texttt{LIMIT 10 OFFSET 10}, pulando a primeira dezena. A lógica se repete para a terceira página com \texttt{LIMIT 10 OFFSET 20}, criando um escoamento suave e escalável que evita carregar toda a tabela na memória RAM do servidor de aplicação.

\begin{quadro_dica}
Sempre que usar \texttt{LIMIT} e \texttt{OFFSET}, use também \texttt{ORDER BY}. Caso contrário, a noção de “primeiras” ou “próximas” linhas perde sentido lógico.
\end{quadro_dica}

\section{DISTINCT}

A cláusula \texttt{DISTINCT} elimina duplicidades do resultado.

\begin{sqlcode}
SELECT DISTINCT curso
FROM alunos;
\end{sqlcode}

Se cem alunos estiverem distribuídos em apenas três cursos, o resultado trará apenas três linhas, uma para cada curso distinto.

\subsection{DISTINCT em mais de uma coluna}

\begin{sqlcode}
SELECT DISTINCT curso, status
FROM alunos;
\end{sqlcode}

Nesse caso, a eliminação de duplicidades ocorre sobre a combinação dos valores de \texttt{curso} e \texttt{status}.

\section{UNION}

O operador \texttt{UNION} combina os resultados de duas consultas compatíveis, eliminando duplicidades entre elas.

\begin{sqlcode}
SELECT nome
FROM alunos
UNION
SELECT nome
FROM professores;
\end{sqlcode}

Do ponto de vista estrutural, para que o banco aprove e processe a operação matemática de união de conjuntos, existem exigências estritas: os dois blocos \texttt{SELECT} envolvidos devem devolver, obrigatoriamente, a exata mesma quantidade posicional de colunas, e as colunas geometricamente alinhadas precisam possuir tipos de dados estritamente compatíveis (não é possível empilhar uma coluna de datas sobre uma coluna numérico-textual incompatível).

\section{UNION ALL}

O \texttt{UNION ALL} também combina resultados, mas preserva todas as linhas, inclusive as repetidas.

\begin{sqlcode}
SELECT nome, 'ALUNO' AS origem
FROM alunos
UNION ALL
SELECT nome, 'PROFESSOR' AS origem
FROM professores;
\end{sqlcode}

O uso deliberado do operador que preserva repetições torna-se vital quando as duplicidades semânticas importam estatisticamente (por exemplo, analisar transações corriqueiras de valores repetidos). Mais importante ainda, o \texttt{UNION ALL} apresenta um ganho colossal de performance em comparação à sua versão sem o "ALL", pois ele instrui o motor do banco a pular a custosa fase de ordenação e varredura de deduplicação, simplesmente empilhando os blocos crus. Para não perder o contexto da origem da informação, uma prática excelente é injetar colunas textuais literais (como \texttt{'ALUNO'} ou \texttt{'PROFESSOR'}), rotulando cada registro durante a fusão.

\section{HAVING revisitado}

Embora já estudado anteriormente, o \texttt{HAVING} se conecta naturalmente ao refinamento das consultas.

\begin{sqlcode}
SELECT curso_id, COUNT(*) AS total
FROM alunos
GROUP BY curso_id
HAVING COUNT(*) >= 3
ORDER BY total DESC;
\end{sqlcode}

Aqui, primeiro os dados são agrupados, depois os grupos são filtrados e, por fim, ordenados.

\section{Funções de janela}

As funções de janela (\textit{window functions}) permitem calcular valores analíticos sobre um conjunto de linhas relacionadas à linha atual, mas sem colapsar o resultado como ocorre com \texttt{GROUP BY}.

\subsection{Média por grupo em cada linha}

\begin{sqlcode}
SELECT
    aluno_id,
    nome,
    curso_id,
    idade,
    AVG(idade) OVER (PARTITION BY curso_id) AS media_idade_curso
FROM alunos;
\end{sqlcode}

Cada linha continua representando um aluno, mas agora traz ao lado a média de idade do seu curso.

\subsection{Classificação por grupo}

\begin{sqlcode}
SELECT
    nome,
    curso_id,
    idade,
    RANK() OVER (PARTITION BY curso_id ORDER BY idade DESC) AS posicao
FROM alunos;
\end{sqlcode}

Essa consulta classifica os alunos dentro de cada curso, do mais velho para o mais novo.

\subsection{Contagem por grupo sem perder linhas}

\begin{sqlcode}
SELECT
    nome,
    curso_id,
    COUNT(*) OVER (PARTITION BY curso_id) AS total_no_curso
FROM alunos;
\end{sqlcode}

\section{Diferença entre GROUP BY e funções de janela}

A distinção conceitual e matemática entre as duas abordagens é central na engenharia de consultas analíticas. Enquanto o \texttt{GROUP BY} atua achatando destrutivamente o conjunto de resultados, resumindo implacavelmente dezenas de linhas originais em uma única e consolidada linha agregada, as modernas funções de janela trabalham preservando intocável cada linha individual do conjunto, injetando elegantemente um cálculo de contexto macro (a estatística do "todo") em uma coluna lateral iterativa.

Essa diferença torna as funções de janela especialmente úteis para relatórios em que se deseja manter o detalhe sem abrir mão da comparação com o grupo.

\section{Views}

\subsection{O que é uma view}

Uma \texttt{VIEW} é uma tabela virtual baseada em uma consulta. Ela não precisa armazenar os dados como uma tabela comum; em vez disso, armazena a lógica que produz aquele resultado.

\subsection{Arquitetura e Boas Práticas: Por que usar Views}

No ecossistema corporativo, o acesso direto a pesadas tabelas físicas por sistemas cliente (como aplicações, APIs ou painéis executivos) é massivamente considerado uma \textbf{má prática arquitetural}. Esse acoplamento direto gera uma enorme repetição de longos trechos SQL nos códigos-fonte, engessa as refatorações estruturais do banco (já que qualquer alteração no esquema ou exclusão de coluna pode quebrar as integrações vitais de forma irreversível) e expõe a topografia crua dos dados sem camadas lógicas de controle de concorrência.

Em oposição a esse caos técnico, a literatura acadêmica e o mercado estipulam o uso mandatório de \texttt{VIEWS} para expor dados. Elas atuam como membranas estabilizadoras blindando o esquema físico: uma View pode abrigar internamente uma malha complexa de mais de dez \texttt{JOINs} assustadores e transformá-la visualmente em uma singela tabela flat virtual, facilitando imediatamente a escrita de relatórios por analistas menos experientes. No futuro, caso o DBA precise modificar tabelas, separar entidades ou criar históricos, ele precisará apenas atualizar o interior da consulta DQL dentro daquela View. Todas as aplicações matrizes continuarão injetando o tradicional e inofensivo \texttt{SELECT * FROM vw\_financeiro}, extraindo os dados exatos sem jamais perceber as engrenagens substituídas nos bastidores. Assim, as Views formam a fundação do isolamento e do encapsulamento de banco de dados moderno.

\subsection{Criando uma view simples}

\begin{sqlcode}
CREATE VIEW vw_alunos_cursos AS
SELECT
    a.nome AS aluno,
    c.nome AS curso
FROM alunos a
INNER JOIN cursos c
    ON a.curso_id = c.curso_id;
\end{sqlcode}

\subsection{Consultando a view}

\begin{sqlcode}
SELECT *
FROM vw_alunos_cursos
ORDER BY aluno;
\end{sqlcode}

\section{Views como abstração}

Uma das principais vantagens das views é abstrair a complexidade estrutural. Em vez de obrigar o usuário a lembrar e reescrever junções repetidas, a lógica já fica encapsulada.

\begin{observacao}
Uma view é especialmente útil quando a mesma consulta passa a ser utilizada repetidamente em exercícios, relatórios, painéis ou rotinas da aplicação.
\end{observacao}

\section{Views e segurança}

As views também podem ser usadas para restringir o que será exposto a determinados usuários. Em vez de conceder acesso direto à tabela original, pode-se conceder acesso apenas à view, que mostra um recorte controlado das informações.

Por exemplo, em um esquema de recursos humanos, uma tabela original contendo dados gerais dos funcionários possui também colunas financeiras sensíveis (como salários e bônus) e dados sigilosos (como laudos médicos). Ao invés de lidar com permissões intrincadas de nível de coluna, o DBA cria uma \texttt{vw\_funcionarios\_publico} excluindo sumariamente as colunas sigilosas da cláusula \texttt{SELECT}. Dessa forma, ocultam-se magicamente dados administrativos, senhas e informações pessoais, fornecendo um túnel limpo e seguro para os analistas trabalharem sem violarem políticas de segurança da informação.

\section{Exemplo resolvido 1 -- Consulta ordenada e paginada}

Suponha que a secretaria queira exibir uma lista de alunos em ordem alfabética, mostrando apenas os dez primeiros de cada página.

\begin{sqlcode}
SELECT nome, curso_id
FROM alunos
ORDER BY nome
LIMIT 10 OFFSET 0;
\end{sqlcode}

\textbf{Resultado Simulado:}

\begin{table}[H]
    \centering
    \begin{tabular}{|l|c|}
    \hline
    \textbf{nome} & \textbf{curso\_id} \\
    \hline
    Amanda Castro & 3 \\
    Bruno Oliveira & 1 \\
    Carlos Luz & 2 \\
    \multicolumn{2}{|c|}{\textit{(... mais 7 linhas)}} \\
    \hline
    \end{tabular}
    \caption{Tabela de saída (Result Set) da paginação inicial.}
\end{table}

A interpretação desta matriz de resultados é direta: os dados foram extraídos e organizados rigorosamente em ordem alfabética. O motor do SGBD cortou a execução e o envio de pacotes pela rede assim que atingiu a marca de dez linhas exatas (\texttt{LIMIT 10}), começando a contar desde a primeira linha absoluta (\texttt{OFFSET 0}), o que traduz a implementação perfeita da tela inicial de um relatório paginado.

\section{Exemplo resolvido 2 -- Lista de cursos distintos}

Suponha que se deseje apenas saber quais cursos aparecem entre os alunos cadastrados.

\begin{sqlcode}
SELECT DISTINCT curso_id
FROM alunos
ORDER BY curso_id;
\end{sqlcode}

\textbf{Resultado Simulado:}

\begin{table}[H]
    \centering
    \begin{tabular}{|c|}
    \hline
    \textbf{curso\_id} \\
    \hline
    1 \\
    2 \\
    3 \\
    \hline
    \end{tabular}
    \caption{Tabela de saída contendo apenas cursos únicos.}
\end{table}

A consulta gerou uma matriz final de apenas uma coluna. O processo eliminou cirurgicamente todas as centenas de repetições do mesmo \texttt{curso\_id} herdadas do cadastro volumoso de alunos. Esse retorno enxuto é classicamente utilizado no preenchimento de código \textit{front-end} para popular as opções de filtros combinados e caixas de seleção em interfaces visuais e aplicativos.

\section{Exemplo resolvido 3 -- União de perfis}

Imagine um relatório institucional que deva exibir nomes de alunos e professores em uma única listagem.

\begin{sqlcode}
SELECT nome, 'ALUNO' AS origem
FROM alunos
UNION ALL
SELECT nome, 'PROFESSOR' AS origem
FROM professores;
\end{sqlcode}

\textbf{Resultado Simulado:}

\begin{table}[H]
    \centering
    \begin{tabular}{|l|c|}
    \hline
    \textbf{nome} & \textbf{origem} \\
    \hline
    Juliana Paes & ALUNO \\
    Lucas Silva & ALUNO \\
    Roberto Gomes & PROFESSOR \\
    Marta Ferreira & PROFESSOR \\
    \hline
    \end{tabular}
    \caption{Tabela consolidada demonstrando a união semântica.}
\end{table}

Para obter esse relatório misto, as duas consultas tiveram que obedecer rigorosamente à mesma estrutura geométrica (uma coluna \texttt{nome} e uma coluna \texttt{origem}). A inclusão dos campos literais injetados manualmente permite ao analista distinguir perfeitamente os papéis institucionais, enquanto o \texttt{UNION ALL} garante um tempo de resposta baixíssimo ao abdicar de deduplicações custosas e focar apenas no empilhamento vertical rápido dos dados na memória RAM.

\section{Exemplo resolvido 4 -- Média do grupo em cada linha}

Deseja-se listar alunos com a média de idade do curso ao lado.

\begin{sqlcode}
SELECT
    nome,
    curso_id,
    idade,
    AVG(idade) OVER (PARTITION BY curso_id) AS media_curso
FROM alunos;
\end{sqlcode}

\textbf{Resultado Simulado:}

\begin{table}[H]
    \centering
    \begin{tabular}{|l|c|c|c|}
    \hline
    \textbf{nome} & \textbf{curso\_id} & \textbf{idade} & \textbf{media\_curso} \\
    \hline
    Ana Lúcia & 1 & 20 & 21.5 \\
    Paulo Freire & 1 & 23 & 21.5 \\
    Luiza Melo & 2 & 19 & 19.0 \\
    \hline
    \end{tabular}
    \caption{Tabela com preservação granular e injeção analítica em coluna adjacente.}
\end{table}

Diferente do agrupamento destrutivo clássico promovido pelo \texttt{GROUP BY}, a tabela final comprova que cada aluno individual e seus detalhes foram integralmente preservados. A função agregada, embalada pela cláusula \texttt{OVER}, atuou calculando a média separadamente para o subgrupo de pertencimento (\texttt{PARTITION BY curso\_id}) e replicou esse mesmo cálculo matricial como um metadado analítico complementar anexado na última coluna de cada respectivo aluno, enriquecendo formidavelmente a capacidade de comparação visual em planilhas de *Business Intelligence*.

\section{Exemplo resolvido 5 -- View para relatório acadêmico}

\begin{sqlcode}
CREATE VIEW vw_notas_alunos AS
SELECT
    a.nome AS aluno,
    d.nome AS disciplina,
    n.nota
FROM notas n
INNER JOIN alunos a
    ON n.aluno_id = a.aluno_id
INNER JOIN disciplinas d
    ON n.disciplina_id = d.disciplina_id;
\end{sqlcode}

\begin{sqlcode}
SELECT *
FROM vw_notas_alunos
ORDER BY aluno, disciplina;
\end{sqlcode}

\textbf{Interpretação do ecossistema e facilitação do código:}

Neste fluxo robusto, o pesadelo lógico das junções multiplicadas (\textit{JOINs}) e da navegação técnica minuciosa através de chaves estrangeiras transversais fica trancado permanentemente dentro da declaração DDL inicial da \texttt{VIEW}. Daquele instante criativo em diante, os usuários finais (ou até microsserviços) interagem unicamente com a fachada virtual \texttt{vw\_notas\_alunos}, inserindo seleções ingênuas como se estivessem trabalhando em cima de uma tabela bruta desnormalizada. Essa separação de fronteiras eleva dramaticamente a manutenibilidade do sistema, previne a duplicação de lógicas de junção por toda a equipe e simplifica absurdamente a confecção de rotinas de auditoria relacional.

\section{Erro comum 1 -- Usar LIMIT sem ORDER BY}

Exemplo problemático:

\begin{sqlcode}
SELECT *
FROM alunos
LIMIT 10;
\end{sqlcode}

Sem ordenação explícita, o conjunto de “dez primeiros” pode não fazer sentido lógico do ponto de vista do relatório.

\section{Erro comum 2 -- Confundir DISTINCT com GROUP BY}

\texttt{DISTINCT} elimina duplicidades. \texttt{GROUP BY} organiza linhas em grupos para permitir agregações. Eles não são equivalentes.

\section{Erro comum 3 -- Usar UNION quando UNION ALL seria mais adequado}

\texttt{UNION} elimina duplicidades e pode custar mais. Se as repetições não forem problema, \texttt{UNION ALL} pode ser a melhor escolha.

\section{Erro comum 4 -- Tentar substituir toda agregação por função de janela}

Funções de janela não substituem todos os usos de \texttt{GROUP BY}. Elas têm papel complementar, não absoluto.

\section{Erro comum 5 -- Criar view sem objetivo claro}

Uma view deve ter propósito claro: simplificação, segurança, relatório ou reutilização. Criar views sem critério pode apenas aumentar a complexidade do esquema.

\section{Boas práticas}

O cientista de dados ou desenvolvedor compreende rapidamente que o \texttt{SELECT} não atende apenas requisitos funcionais de negócio, mas responde diretamente por requisitos não funcionais de extrema criticidade (segurança e desempenho). Torna-se mandatório o acoplamento do \texttt{ORDER BY} a instruções restritivas de limite de busca para evitar o retorno acidental e caótico de tabelas volumosas inteiras. A instrução \texttt{DISTINCT} jamais deve ser utilizada como um escudo desleixado para esconder e silenciar consultas defeituosas contendo duplicação acidental (pois o \textit{Distinct} força um custo computacional massivo de processamento topológico no SGBD). Do mesmo modo prático, prefere-se quase integralmente o uso de \texttt{UNION ALL} ao \texttt{UNION} nativo sempre que a chance de uma duplicidade inter-resultados for matematicamente improvável. Para manter o código imune a dívidas técnicas, as \textit{Views} lógicas devem incorporar invariavelmente nomenclaturas padronizadas, e as ferramentas analíticas baseadas em \textit{Window Functions} devem dominar as escolhas de modelagem quando a devolução gráfica simultânea das linhas granulares for inegociável.

\begin{quadro_dica}
Uma convenção útil é prefixar views com \texttt{vw\_}. Isso ajuda a distingui-las rapidamente de tabelas comuns.
\end{quadro_dica}

\section{Minicaso integrador -- Painel acadêmico resumido}

Considere o cenário real no qual a coordenação pedagógica deseja estruturar o \textit{dashboard} (painel de controle) central da instituição. Para abastecer a interface gráfica, a camada de acesso a dados deverá prover sequencialmente: uma listagem paginada de alunos para a tela inicial, a extração de cursos únicos para compor a caixa de seleção (\textit{combobox}) de filtros de busca, um relatório unificado empilhando nomes de alunos e professores para alimentar a barra de pesquisa global, a exibição analítica da média de idade da turma alocada lado a lado ao perfil de cada aluno e, por fim, a criação de uma fundação arquitetural sólida baseada em \textit{Views} para suportar a geração do pesado relatório estatístico de notas.

Uma sequência contínua de blocos DQL para suprir essas exigências de negócio seria:

\subsection*{Passo 1 -- Lista paginada}

\begin{sqlcode}
SELECT nome, curso_id
FROM alunos
ORDER BY nome
LIMIT 10 OFFSET 0;
\end{sqlcode}

\subsection*{Passo 2 -- Cursos distintos}

\begin{sqlcode}
SELECT DISTINCT curso_id
FROM alunos
ORDER BY curso_id;
\end{sqlcode}

\subsection*{Passo 3 -- União de nomes}

\begin{sqlcode}
SELECT nome, 'ALUNO' AS origem
FROM alunos
UNION ALL
SELECT nome, 'PROFESSOR' AS origem
FROM professores;
\end{sqlcode}

\subsection*{Passo 4 -- Média por curso em cada linha}

\begin{sqlcode}
SELECT
    nome,
    curso_id,
    idade,
    AVG(idade) OVER (PARTITION BY curso_id) AS media_curso
FROM alunos;
\end{sqlcode}

\subsection*{Passo 5 -- View de relatório}

\begin{sqlcode}
CREATE VIEW vw_notas_alunos AS
SELECT
    a.nome AS aluno,
    d.nome AS disciplina,
    n.nota
FROM notas n
INNER JOIN alunos a
    ON n.aluno_id = a.aluno_id
INNER JOIN disciplinas d
    ON n.disciplina_id = d.disciplina_id;
\end{sqlcode}

\begin{sqlcode}
SELECT *
FROM vw_notas_alunos
ORDER BY aluno, disciplina;
\end{sqlcode}

\textbf{Comentário didático:}
Esse minicaso mostra como vários recursos de refinamento podem coexistir em um mesmo cenário. A consulta deixa de ser apenas mecanismo de busca e passa a atuar como ferramenta de apresentação, organização e reutilização.
